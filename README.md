# Адресное хранение склада

Веб-приложение для адресного хранения на складе: сервер на Node.js/Express с
базой данных SQLite и фронтенд (одна HTML-страница), который ходит в REST
API. Все, кто открывает адрес сайта, видят одни и те же данные из общей базы,
а не локальную копию в браузере.

---

## Руководство пользователя

Этот раздел — не про код, а про то, как пользоваться сайтом, если вы видите
его первый раз. Читайте по своей задаче, всё остальное не обязательно знать.

### Шапка страницы (сверху, видна всегда)

- **Адресное хранение · подключение…/localStorage/сервер** — надпись рядом
  с названием показывает, откуда сейчас взяты данные (обычно «сервер» —
  значит всё в порядке, все видят одно и то же).
- **Цифры справа** («шт всего», «артикулов», «ячеек занято», «строк в служ.
  зонах») — это просто счётчики по всей базе, обновляются сами.
- **⟳ Синхронизировать** — нажмите, если думаете, что коллега только что
  что-то поменял, а у вас на экране ещё старые данные. Страница и так сама
  подтягивает свежие данные раз в минуту, эта кнопка — просто «сделать это
  прямо сейчас».
- **↺ Отменить** — откатывает самое последнее действие (своё или чужое):
  удалили не ту запись, перепутали ячейку при переносе, случайно обменяли
  местами не те ряды — жмите «Отменить», и всё вернётся как было. Эта кнопка
  в шапке всегда откатывает только одно, самое последнее действие; не
  помнит про загрузку нового файла .xlsx целиком.
- **📋 Журнал** — список того, что вообще происходило на складе за последние
  14 дней: кто/что добавил, удалил, перенёс, переименовал. Полезно, если
  нужно понять «а куда делся товар». У каждой строки, которую можно
  отменить, есть своя кнопка **↺ Отменить** — то есть не обязательно
  откатывать всё подряд от самого свежего действия: можно найти в списке
  нужную запись (например, трёхдневной давности) и отменить именно её. Если
  выбранная запись — не самая последняя в журнале, сайт сначала спросит
  подтверждение и предупредит, что после неё уже были другие изменения (на
  случай, если они затрагивали те же самые записи).
- **🌙 / ☀️** — переключает тёмную/светлую тему оформления, чисто по вкусу.
- **База данных ▾** — тут два действия:
  - **⬆ Загрузить новые данные (.xlsx)** — полностью заменяет весь сток на
    сервере тем, что в вашем Excel-файле. Это **не добавление**, а замена
    целиком — используйте, когда нужно обновить остатки по свежей выгрузке
    из 1С или похожей системы. ABC-классы и служебные зоны при этом не
    трогаются.
  - **⬇ Экспорт в Excel** — скачивает текущее состояние склада (то, что
    сейчас в базе на сервере) в виде .xlsx-файла — удобно для отчёта или
    архива.

### Вкладки

Под шапкой — четыре вкладки, между которыми можно переключаться в любой
момент, ничего не потеряется:

1. **Схема склада** — визуальная карта: ряды → стеллажи → ярусы.
2. **Таблица данных** — весь сток одним списком, с фильтрами и правкой.
3. **Служебные зоны** — товары не на полке, а в отдельных статусах (брак,
   карантин, приёмка и т.п.).
4. **Рекомендация пикинга** — подсказки, где по-хорошему должен лежать
   каждый артикул (по ABC-классу).

### Вкладка «Схема склада»

- **Чипы рядов** (кнопки «Ряд 01», «Ряд 02»…) — клик переключает, какой ряд
  сейчас показан на схеме ниже. Цифра рядом — сколько ячеек занято в этом
  ряду.
- **Поиск по схеме** (строка с лупой) — введите артикул, номер ячейки или
  код ТЕ и нажмите Enter — сайт сам переключится на нужный ряд и подсветит
  ячейку мигающей рамкой.
- **Иконка штрихкода рядом с поиском** — открывает поиск товара по
  штрихкоду через камеру телефона/ноутбука (см. раздел про сканер ниже).
- **✥ Режим перемещения** — на телефоне мышью перетаскивать ничего нельзя,
  поэтому есть отдельный «режим для пальца»: включаете кнопку, касаетесь
  одной ячейки/ряда/стеллажа, потом другой — они меняются местами. Кнопка
  видна только на телефонах и планшетах — на компьютере с мышью она скрыта,
  там всё двигается обычным перетаскиванием (drag-and-drop).
- **⚙ Управление рядом** — открывает окно с настройками текущего ряда:
  - переименовать ряд (изменится код у всех записей автоматически);
  - изменить список **стеллажей** — добавить новый номер, убрать пустой,
    перетащить, чтобы поменять порядок отображения;
  - изменить список **ярусов** (это те самые «строки» по высоте стеллажа —
    01, 02, 03… и специальные A1/B1) — точно так же можно добавить ярус из
    выпадающего списка, убрать пустой или перетащить, чтобы задать порядок;
  - **Удалить ряд** — удаляет ряд целиком, но только если в нём совсем нет
    товара (иначе сайт откажет и объяснит, где именно ещё есть остаток).
- **☑ Выбрать ряды** → **🗑 Удалить выбранные** — если нужно снести сразу
  несколько пустых рядов, не заходя в «Управление рядом» по одному:
  нажмите «Выбрать ряды», кликните по нужным чипам рядов (они станут
  красными), затем «Удалить выбранные (N)». Ряды, где ещё остался товар,
  просто не удалятся — сайт покажет, какие именно и почему.
- **Перетаскивание** (на компьютере): перетащите один ряд на другой чип —
  весь товар обоих рядов поменяется местами; то же самое для стеллажей и
  отдельных ячеек внутри ряда.
- **+ Добавить ряд** — создаёт новый ряд с нуля: указываете номер и диапазон
  стеллажей, дальше состав и порядок можно поправить через «Управление
  рядом».
- **Легенда под схемой** — подсказка, что означает цвет и обводка ячейки
  (класс ABC, наличие резерва на ярусах выше первого, и т.д.).

### Сканер штрихкодов — как это работает

Кнопка со значком штрихкода (▤) открывает окно сканера. Есть два способа
им пользоваться:

1. **Через камеру** — наведите камеру телефона/ноутбука на штрихкод, сайт
   сам распознает код (поддерживаются EAN-13, EAN-8, UPC-A, UPC-E, Code128,
   Code39, ITF, QR) и сразу покажет найденный товар.
2. **Вручную** — если камеры нет или код не читается, можно просто ввести
   код (или артикул, или номер ячейки) в поле и нажать «Найти».

**Если вы пользуетесь отдельным ТСД (терминалом сбора данных)** — это
физический сканер-«клавиатура»: он не открывает окно камеры, а просто
печатает считанный код в то поле, где сейчас курсор (например, в поиск по
схеме), и нажимает Enter сам. Сайт понимает, что разные типы штрихкода
(EAN-13, UPC-A, GTIN-14 и т.п.) могут кодировать один и тот же товар
по-разному — например, с лишним нулём в начале — и проверяет все похожие
варианты кода, а не только точное совпадение буква-в-букву. Если ваш ТСД
настроен слать вместо Enter клавишу Tab — сообщите, это отдельная
настройка, которую можно добавить.

### Вкладка «Таблица данных»

- **Поиск и фильтры сверху** — по артикулу/названию, по категории, по
  ABC-классу.
- **Правка прямо в таблице** — кликните по остатку или ячейке в строке,
  введите новое значение — сохранится сразу же на сервере. Если сервер
  вернул ошибку (например, нет связи), значение само откатится обратно, и
  появится сообщение, что не так.
- **🗑 (корзина в конце строки)** — удаляет одну запись.
- **Добавить товар** — кнопка над таблицей открывает форму: ячейка,
  артикул, наименование, остаток, даты, код ТЕ.
- **Мультивыбор строк** (галочки слева) — отметьте несколько записей и
  перенесите их все разом в другую ячейку/зону, либо удалите все разом —
  не нужно делать это по одной.

### Вкладка «Служебные зоны»

Здесь товар, который сейчас не стоит на обычной полке, а находится в
отдельном статусе — карантин, брак/возврат, приёмка и т.п. Каждая зона —
физически отдельное место, чтобы такой товар не путался с обычным стоком:

- **Создать зону** — задаёте название, при необходимости ставите галочку
  «изолированная».
- **Переименовать / переключить изоляцию** — через карточку зоны.
- **Удалить зону** — либо только если она пустая, либо принудительно —
  вместе со всем, что в ней лежит (сайт явно предупредит).
- Перенос товара в зону делается там же, где и любой другой массовый
  перенос — через мультивыбор в таблице данных.

### Вкладка «Рекомендация пикинга»

Показывает, на каком ярусе по-хорошему должен лежать каждый артикул, исходя
из его ABC-класса (частые/ходовые товары — на ярус 1, чтобы не наклоняться
и не подниматься каждый раз), а какие ячейки — под резервное пополнение
(ярусы выше первого). Можно фильтровать по классу ABC и категории, искать
конкретный артикул.

### Пользуетесь с телефона?

Интерфейс отдельно оптимизирован под мобильный экран: крупные кнопки под
палец, поля ввода не вызывают непроизвольный зум на iPhone, карточки и
модальные окна на маленьком экране открываются на весь экран, а схема
склада и режим «✥ Режим перемещения» заточены под управление одним пальцем
вместо мышки.

## Репозиторий на GitHub

Код проекта лежит на GitHub: **github.com/xsibn/storage**.

### Скачать проект

```bash
git clone https://github.com/xsibn/storage.git
cd storage
npm install
npm start
```

Либо без git — на странице репозитория кнопка **Code ▾ → Download ZIP**
скачивает архив с текущим содержимым ветки `main`.

### Обновить сервер до последней версии из репозитория

Если сервер уже развёрнут и просто нужно подтянуть свежие изменения:

```bash
cd storage
git pull
npm install   # если менялись зависимости в package.json
pm2 restart warehouse   # или ваш способ перезапуска, см. раздел «Запуск как службы»
```

`git pull` не трогает `data/warehouse.db` — база и её содержимое в
репозиторий не входят (см. `.gitignore`), так что обновление кода никогда
не затирает текущий сток.

### Внести и сохранить свои изменения

```bash
git add .
git commit -m "Что именно поменяли"
git push
```

Если правите код прямо на боевом сервере (например, через SSH), удобнее
сначала закоммитить и запушить оттуда же — тогда GitHub всегда отражает то,
что реально сейчас работает.

### Структура репозитория

Совпадает с разделом «Что внутри» выше: `server.js`, `db.js`, `public/`,
`seed/` — в репозиторий. Папка `data/` (сама база SQLite) и `node_modules/`
в репозиторий не попадают — они создаются/ставятся заново на каждом
сервере командами `npm install` и первым запуском `npm start`.

---



```
warehouse-server/
├── server.js              — Express-сервер, все маршруты REST API, разбор .xlsx при импорте
├── db.js                  — вся работа с SQLite: схема, запросы, бизнес-логика, журнал изменений
├── package.json
├── seed/
│   ├── seed.json           — исходный сток (загружается один раз при первом запуске, если база пустая)
│   └── abc-classes.json    — фиксированная ABC-классификация артикулов (постоянная, задаётся файлом)
├── data/                   — здесь появится файл базы warehouse.db (создаётся сам при первом запуске)
└── public/
    ├── index.html          — сама страница (разметка + стили)
    └── app.js               — вся логика фронтенда, ходит в API
```

## Установка и запуск

Нужен Node.js версии 18 или новее.

```bash
cd warehouse-server
npm install
npm start
```

По умолчанию сервер слушает порт **3000**. Открыть `http://ваш-сервер:3000`.

Порт можно поменять переменной окружения:

```bash
PORT=8080 npm start
```

Для разработки есть `npm run dev` — запускает сервер через `node --watch`,
он сам перезапускается при изменении файлов.

При первом запуске, если база пустая, сервер сам загрузит в неё данные из
`seed/seed.json`. Дальше база живёт самостоятельно — при перезапуске сервера
seed повторно не применяется, пока таблица не пустая. Заодно при каждом
старте сервер:
- «расширяет» сохранённую структуру склада (ряды/стеллажи/ярусы) до полного
  охвата seed-файла — это чинит ситуацию, когда ряд или стеллаж «пропадал» с
  карты склада после того, как становился полностью пустым;
- сверяет структуру склада с тем, что реально есть в таблице записей сейчас,
  и чинит расхождения, оставшиеся от операций с рядами/стеллажами, сделанных
  до появления этой защиты;
- регистрирует уже существующие в данных служебные зоны (Карантин, Приёмка
  и т.п.) как управляемые зоны.

Если на сервере нет компилятора и установка `better-sqlite3` падает с ошибкой
сборки, поставьте инструменты сборки и повторите `npm install`:

```bash
# Debian/Ubuntu
sudo apt-get update && sudo apt-get install -y build-essential python3
```

## Возможности

### Таблица данных
- Просмотр всего стока: категория, артикул, наименование, ABC-класс, остаток,
  список ячеек, где хранится товар.
- Ручная правка остатка и/или ячейки прямо в таблице — сохраняется сразу
  через API; при ошибке значение откатывается обратно и показывается
  сообщение об ошибке.
- Добавление нового товара в ячейку и удаление ошибочно добавленной записи.
- Мультивыбор строк и массовые операции: перенос сразу нескольких записей в
  другую ячейку/зону или массовое удаление.

### Карта склада (ряды, стеллажи, ярусы)
- Создание нового ряда с нуля (номер ряда, список стеллажей, ярусы).
- Удаление ряда целиком (только если он пустой), в том числе **массовое**
  удаление сразу нескольких рядов через режим множественного выбора чипов
  рядов — каждый ряд по-прежнему проверяется независимо, и невозможно
  случайно удалить ряд, где ещё есть товар.
- Переименование ряда — все записи автоматически переносятся под новый код.
- Изменение состава стеллажей ряда (добавление/удаление ячеек) и
  пользовательский порядок стеллажей — перетаскиванием или стрелками ◀▶.
- Изменение состава **ярусов** ряда тем же способом (добавление/удаление
  из фиксированного набора 01–08, A1, B1) и их порядка — то есть можно
  менять не только «сколько стеллажей», но и «сколько строк по высоте» у
  ряда, прямо в окне «Управление рядом».
- Обмен местами содержимого двух рядов целиком или двух стеллажей внутри
  одного ряда — удобно для физической перестановки товара на складе без
  ручного переноса каждой позиции.

### Служебные зоны
Помимо адресных ячеек в базе можно вести произвольные служебные зоны
(например, «Карантин», «Приёмка», «Брак»):
- создание зоны с опциональной пометкой «изолированная» (`isolate`);
- переименование зоны и переключение изоляции;
- удаление пустой зоны либо принудительное удаление вместе с содержимым;
- перенос выбранных записей в зону через массовую операцию.

### ABC-классификация
- Классы (A/B/C) по каждому артикулу задаются файлом `seed/abc-classes.json`
  и хранятся в базе как фиксированный справочник, не завязанный на импорт
  стока.
- На странице — сводка по ABC-классам, фильтрация таблицы по классу,
  рекомендации по пикинг-адресам (ярус 1) и адресам пополнения (резерв) с
  учётом доли объёма стока по каждому артикулу.

### Журнал изменений и отмена действий
- Все изменяющие операции (создание/правка/удаление записи, обмен рядов и
  стеллажей, переименование ряда/зоны, изменение состава стеллажей и ярусов,
  создание/удаление ряда, массовые перенос/удаление) пишутся в журнал
  активности.
- Журнал хранит записи за последние 14 дней, более старые удаляются
  автоматически при каждой новой записи.
- Кнопка «↺ Отменить» в шапке откатывает последнее действие из журнала,
  проигрывая обратную операцию (отмена массового переноса вернёт каждую
  запись в её исходную ячейку, отмена удаления — восстановит запись и т.д.).
- В окне журнала у каждой отменяемой записи есть собственная кнопка
  «↺ Отменить» — можно отменить не только самое последнее действие, а любую
  конкретную запись из истории (`POST /api/activity/:id/undo`), той же самой
  обратной операцией. Если это не самая свежая запись в журнале, фронтенд
  показывает предупреждение перед подтверждением — сервер не умеет
  автоматически определять конфликт с более поздними изменениями тех же
  записей, поэтому ответственность за смысл такой отмены на пользователе.
- Импорт нового файла целиком (полная замена базы) под отмену не подпадает
  специально — хранить снимок всей базы под каждую загрузку избыточно.

### Импорт и экспорт
- «Загрузить новые данные» — `.xlsx`-файл целиком отправляется на сервер,
  который сам его разбирает и **полностью заменяет** содержимое базы (кроме
  ABC-классов и служебных зон — они не зависят от импорта).
  - Сервер умеет искать настоящую шапку таблицы даже если перед ней есть
    служебные строки отчёта (типично для выгрузок «1С: Отчет по остаткам») —
    ищет строку с колонками «Ячейка» и «Артикул», а не считает шапкой
    обязательно первую строку листа.
  - Колонки сопоставляются по названию независимо от порядка и небольших
    отличий в написании («Артикул» / «Артикул » с пробелом и т.п.):
    Ячейка, Артикул, Наименование, Остаток, Дата изготовления, Срок годности, ТЕ.
- «Экспорт в Excel» — собирает `.xlsx` из текущего состояния базы на сервере
  (а не из того, что успело загрузиться в браузер), включая пометку типа
  строки — «Адресная ячейка» или «Служебная зона».

### Синхронизация между пользователями
- **Хранилище одно** — файл `data/warehouse.db` на сервере. Копий таблицы
  рядом с сайтом или зашитых в HTML данных нет.
- Страница при открытии делает `GET /api/records` и получает актуальный сток
  из базы.
- Кнопка «Синхронизировать» в шапке принудительно подтягивает свежие данные
  (например, если их обновил коллега). Страница также сама подтягивает
  данные раз в минуту в фоне — но не в момент, когда вы редактируете поле или
  открыта карточка ячейки, чтобы не сбить ввод.

### Поиск по штрихкоду и работа с ТСД
- Поиск товара по коду (камера, ручной ввод в окне сканера, поиск по схеме,
  поиск по таблице) сначала пробует точное совпадение с артикулом/ячейкой/ТЕ,
  причём не только буква-в-букву, но и по всем разумным эквивалентным формам
  одного и того же кода: UPC-A (12 цифр) ⇄ EAN-13 (13 цифр, с ведущим нулём)
  ⇄ GTIN-14 (14 цифр, два ведущих нуля), плюс вырезание служебных префиксов,
  которые иногда добавляют сканеры (AIM-идентификатор символики вроде `]C1`,
  обёртка GS1 `(01)…` для GTIN-14). Это нужно, потому что физический
  ТСД-сканер работает как клавиатура — печатает код в текущее поле и жмёт
  Enter — и не сообщает, каким именно способом штрихкод был закодирован; без
  этой нормализации один и тот же товар при сканировании разными типами
  штрихкода мог не находиться.
- Если совпадений по коду не нашлось совсем, поиск падает на менее строгий
  вариант — вхождение подстроки, как и раньше.

## API

| Метод  | Путь                              | Что делает |
|--------|-----------------------------------|------------|
| GET    | `/api/records`                    | Весь текущий сток + метаданные: источник, время импорта, структура склада (layout), ABC-классы, список зон |
| PATCH  | `/api/records/:id`                | Правка одной записи: `{ "qty": 10 }` и/или `{ "cell": "01-12-02" }` |
| POST   | `/api/records`                    | Добавить новую запись: `{ "cell", "article", "name", "qty", "mfg", "exp", "te" }` (`cell` и `article` обязательны) |
| DELETE | `/api/records/:id`                | Удалить запись |
| POST   | `/api/records/bulk-move`          | Массовый перенос: `{ "ids": [1,2,3], "cell": "Карантин" }` |
| POST   | `/api/records/bulk-delete`        | Массовое удаление: `{ "ids": [1,2,3] }` |
| POST   | `/api/records/swap-rows`          | Поменять местами весь товар двух рядов: `{ "rowA": "01", "rowB": "02" }` |
| POST   | `/api/records/swap-racks`         | Поменять местами два стеллажа в одном ряду: `{ "row": "01", "rackA": 1, "rackB": 2 }` |
| POST   | `/api/layout`                     | Создать новый ряд: `{ "row": "05", "racks": [1,2,3], "levels": [...] }` |
| DELETE | `/api/layout/:row`                | Удалить ряд целиком (только пустой) |
| PUT    | `/api/layout/:row/rename`         | Переименовать ряд: `{ "newRow": "07" }` — все записи переносятся автоматически |
| PUT    | `/api/layout/:row/racks`          | Задать список стеллажей ряда: `{ "racks": [1,2,3,4] }` |
| PUT    | `/api/layout/:row/levels`         | Задать список ярусов ряда: `{ "levels": ["01","02","A1"] }` |
| PUT    | `/api/layout/:row/rack-order`     | Сохранить порядок стеллажей ряда: `{ "order": [3,1,2] }` |
| POST   | `/api/zones`                      | Создать служебную зону: `{ "name": "Карантин", "isolate": true }` |
| PATCH  | `/api/zones/:name`                | Переименовать и/или переключить изоляцию: `{ "newName": "...", "isolate": true }` |
| DELETE | `/api/zones/:name`                | Удалить пустую зону (`?force=true` — вместе с содержимым) |
| GET    | `/api/activity`                   | Последние записи журнала изменений (`?limit=200`) |
| POST   | `/api/activity/undo`              | Отменить последнее действие из журнала |
| POST   | `/api/activity/:id/undo`          | Отменить конкретную запись журнала по её id (не обязательно последнюю) |
| POST   | `/api/import`                     | Загрузка нового `.xlsx` (`multipart/form-data`, поле `file`) — заменяет весь сток |
| GET    | `/api/export`                     | Скачать текущее состояние базы как `.xlsx` |
| GET    | `/api/health`                     | Проверка, что сервер жив |

Любой прочий путь (например, прямая ссылка на подстраницу) отдаёт
`public/index.html`, чтобы работали закладки и обновление страницы.

## Запуск как службы (чтобы жил после перезагрузки/отключения SSH)

Проще всего — через `pm2`:

```bash
npm install -g pm2
cd warehouse-server
pm2 start server.js --name warehouse
pm2 save
pm2 startup   # выведет команду для автозапуска при перезагрузке сервера
```

Или через systemd — создайте `/etc/systemd/system/warehouse.service`:

```ini
[Unit]
Description=Адресное хранение склада
After=network.target

[Service]
WorkingDirectory=/путь/до/warehouse-server
ExecStart=/usr/bin/node server.js
Restart=on-failure
Environment=PORT=3000
User=www-data

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now warehouse
```

## Если сайт должен открываться на 80/443 через домен

Заведите Node-сервер на localhost:3000 (как выше) и поставьте перед ним nginx
как обратный прокси:

```nginx
server {
    listen 80;
    server_name склад.вашдомен.ру;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

Для HTTPS проще всего добавить сертификат через `certbot --nginx`.

## Резервные копии

Вся база — один файл `data/warehouse.db` (плюс `-wal`/`-shm` рядом, пока
сервер запущен). Бэкап = скопировать этот файл. Проще всего копировать его,
когда сервер ненадолго остановлен, либо использовать
`sqlite3 data/warehouse.db ".backup copy.db"` на живой базе.

## Известные ограничения этой версии

- Синхронизация не мгновенная — раз в минуту в фоне плюс кнопка вручную,
  без веб-сокетов и live-обновлений «в реальном времени». Для склада с
  редкими правками этого обычно достаточно; если нужна мгновенная
  синхронизация нескольких сборщиков одновременно — можно добавить позже.
- Нет авторизации: любой, у кого есть ссылка на сайт, может редактировать
  данные и загружать новый сток. Если это важно — можно добавить пароль
  или логин отдельным шагом.
- Журнал изменений хранит только последние 14 дней. Отмена (`undo`) доступна
  как для последнего действия, так и для любой конкретной записи журнала по
  отдельности, но не для полной замены базы через импорт `.xlsx`. Отмена
  записи, которая не является самой последней, не проверяет автоматически,
  не затрагивали ли те же данные более поздние действия — сайт только
  предупреждает об этом перед подтверждением.
