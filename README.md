# Адресное хранение склада — сервер с базой данных

Node.js + Express + SQLite (файл `data/warehouse.db`). Сервер отдаёт сам сайт
(папка `public/`) и REST API поверх настоящей базы данных, так что все, кто
открывает адрес сайта, видят одни и те же данные, а правки одного человека
видят остальные (после синхронизации).

## Что внутри

```
warehouse-server/
├── server.js        — Express-сервер и все API-маршруты
├── db.js             — работа с SQLite (схема, запросы)
├── package.json
├── seed/seed.json    — исходный сток на 15.07.2026 (для первого запуска)
├── data/              — тут появится файл базы warehouse.db (создаётся сам)
└── public/
    ├── index.html     — сама страница
    └── app.js          — вся логика фронтенда, ходит в API
```

## Установка и запуск

Нужен Node.js версии 18 или новее.

```bash
cd warehouse-server
npm install
npm start
```

По умолчанию сервер слушает порт **3000**. Открыть `http://ваш-сервер:3000`.

Порт можно поменять переменной окружения:

```bash
PORT=8080 npm start
```

При первом запуске, если база пустая, сервер сам загрузит в неё данные из
`seed/seed.json` (это ваш файл стока на 15.07.2026). Дальше база живёт
самостоятельно — при перезапуске сервера seed повторно не применяется, пока
таблица не пустая.

Если на сервере нет компилятора и установка `better-sqlite3` падает с ошибкой
сборки, поставьте инструменты сборки и повторите `npm install`:

```bash
# Debian/Ubuntu
sudo apt-get update && sudo apt-get install -y build-essential python3
```

## Как теперь синхронизируются данные

- **Хранилище одно** — файл `data/warehouse.db` на сервере. Никакой копии
  таблицы рядом с сайтом хранить не нужно, и в сам HTML ничего не зашито.
- **Просмотр данных**: страница при открытии делает `GET /api/records` и
  получает актуальный сток из базы.
- **Ручная правка** (остаток или ячейка в «Таблице данных») сразу уходит
  запросом `PATCH /api/records/:id` и сохраняется в базе. Если запрос не
  прошёл — значение в интерфейсе откатится назад и покажется ошибка.
- **«Загрузить новые данные»** — файл целиком отправляется на сервер
  (`POST /api/import`), сервер сам разбирает его и **полностью заменяет**
  содержимое базы. Это видят все, кто потом откроет сайт или нажмёт
  «Синхронизировать».
- **«Синхронизировать»** — кнопка в шапке принудительно подтягивает свежие
  данные из базы (например, если их обновил коллега). Страница также сама
  подтягивает данные раз в минуту в фоне — но не во время, когда вы
  редактируете поле или открыта карточка ячейки, чтобы не сбить ввод.
- **«Экспорт в Excel»** — тоже идёт на сервер (`GET /api/export`) и собирает
  файл из текущего состояния базы, а не из того, что успело загрузиться в
  ваш браузер.

## API (если пригодится для интеграций)

| Метод | Путь                | Что делает |
|-------|---------------------|------------|
| GET   | `/api/records`      | Весь сток + метаданные (источник, время последнего импорта) |
| PATCH | `/api/records/:id`  | Правка одной записи: `{ "qty": 10 }` и/или `{ "cell": "01-12-02" }` |
| POST  | `/api/import`       | Загрузка нового `.xlsx` (`multipart/form-data`, поле `file`) — заменяет всю базу |
| GET   | `/api/export`       | Скачать текущее состояние базы как `.xlsx` |
| GET   | `/api/health`       | Проверка, что сервер жив |

## Запуск как службы (чтобы жил после перезагрузки/отключения SSH)

Проще всего — через `pm2`:

```bash
npm install -g pm2
cd warehouse-server
pm2 start server.js --name warehouse
pm2 save
pm2 startup   # выведет команду для автозапуска при перезагрузке сервера
```

Или через systemd — создайте `/etc/systemd/system/warehouse.service`:

```ini
[Unit]
Description=Адресное хранение склада
After=network.target

[Service]
WorkingDirectory=/путь/до/warehouse-server
ExecStart=/usr/bin/node server.js
Restart=on-failure
Environment=PORT=3000
User=www-data

[Install]
WantedBy=multi-user.target
```

```bash
sudo systemctl daemon-reload
sudo systemctl enable --now warehouse
```

## Если сайт должен открываться на 80/443 через домен

Заведите Node-сервер на localhost:3000 (как выше) и поставьте перед ним nginx
как обратный прокси:

```nginx
server {
    listen 80;
    server_name склад.вашдомен.ру;

    location / {
        proxy_pass http://127.0.0.1:3000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}
```

Для HTTPS проще всего добавить сертификат через `certbot --nginx`.

## Резервные копии

Вся база — один файл `data/warehouse.db` (плюс `-wal`/`-shm` рядом, пока
сервер запущен). Бэкап = скопировать этот файл. Проще всего копировать его,
когда сервер ненадолго остановлен, либо использовать `sqlite3 data/warehouse.db ".backup copy.db"`
на живой базе.

## Известные ограничения этой версии

- Синхронизация не мгновенная — раз в минуту в фоне плюс кнопка вручную,
  без веб-сокетов и live-обновлений «в реальном времени». Для склада с
  редкими правками этого обычно достаточно; если нужна мгновенная
  синхронизация нескольких сборщиков одновременно — можно добавить позже.
- Нет авторизации: любой, у кого есть ссылка на сайт, может редактировать
  данные и загружать новый сток. Если это важно — можно добавить пароль
  или логин отдельным шагом.
